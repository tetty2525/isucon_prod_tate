## 2. ISUCON 黄金の改善サイクル（ワークフロー）

このリポジトリでは、作業を爆速で回すための自動化スクリプトが整備されています。
以下の手順でチューニングループを回してください。

1. **コードの変更**
   ローカルの `webapp/python` や `etc/` 配下の設定ファイルを編集する。
2. **変更のコミット（重要）**
   どんな変更をしたかCSVに記録されるため、ベンチ前に必ずコミットする。
   `git commit -am "N+1を解消"`
3. **デプロイとベンチマーク実行**
   ```bash
   make deploy
   make bench
   ```
   ※ `make bench` を実行すると、ベンチマーク中の正確なリソース使用量（CPU、ネットワーク等）が計測され、自動的に `docs/score_history.csv` の末尾に追記されます。
4. **ログの解析（ボトルネック特定）**
   ```bash
   make anal
   ```
   * **alp**: どこのエンドポイント（URL）が遅いか、レスポンスタイムのランキングを出力します。
   * **pt-query-digest**: どのSQLが重いか、DBを圧迫しているクエリのランキングを出力します。

## 3. リアルタイム監視（Netdata）

ベンチマーク実行中の「一瞬のCPUスパイク」や「メモリの枯渇」をリアルタイムで視覚的に監視するための仕組みが導入済みです。

### 起動手順
ローカルの別ターミナルを開き、以下のポートフォワーディングコマンドを実行しっぱなしにします。
```bash
ssh -N -L 19999:127.0.0.1:19999 -i ~/.ssh/ws-default-keypair.pem isucon@54.150.95.245
```
ブラウザで **`http://localhost:19999/`** にアクセスすると監視ダッシュボードが開きます（サインイン不要。一番下の「Skip and use the dashboard anonymously」をクリック）。
ベンチマーク実行中は、画面上部の4つのメーター（CPU、RAM、Network、Disk）を眺めて異常がないか確認してください。

## 5. 次のステップ（Wave 3以降のヒント）
alpの解析結果から、`/image/[0-9]+` という画像配信エンドポイントが圧倒的に時間を消費していることがわかっています。
現在、画像データはDB（`posts` テーブルの `imgdata` カラム）にBLOBとして保存されています。これを**「画像ファイルとしてディスクに書き出し、Nginxから直接配信する」**ようにPythonアプリを改修することで、さらなる大幅なスコアアップが見込めます。
