## 2. ISUCON 黄金の改善サイクル（ワークフロー）

このリポジトリでは、作業を爆速で回すための自動化スクリプトが整備されています。
以下の手順でチューニングループを回してください。

1. **コードの変更**
   ローカルの `webapp/python` や `etc/` 配下の設定ファイルを編集する。
2. **変更のコミット（重要）**
   どんな変更をしたかCSVに記録されるため、ベンチ前に必ずコミットする。
   `git commit -am "N+1を解消"`
3. **デプロイとベンチマーク実行**
   ```bash
   make deploy
   make bench
   ```
   ※ `make bench` を実行すると、ベンチマーク中の正確なリソース使用量（CPU、ネットワーク等）が計測され、自動的に `docs/score_history.csv` の末尾に追記されます。
4. **ログの解析（ボトルネック特定）**
   ```bash
   make anal
   ```
   * **alp**: どこのエンドポイント（URL）が遅いか、レスポンスタイムのランキングを出力します。
   * **pt-query-digest**: どのSQLが重いか、DBを圧迫しているクエリのランキングを出力します。

ブラウザで **`http://localhost:19999/`** にアクセスすると監視ダッシュボードが開きます（サインイン不要。一番下の「Skip and use the dashboard anonymously」をクリック）。
ベンチマーク実行中は、画面上部の4つのメーター（CPU、RAM、Network、Disk）を眺めて異常がないか確認してください。

## 4. これまでの改善ログと現在の状況

### 既に定着している改善
* `unattended-upgrades` を止めて、ベンチ中のブレ要因を減らした。
* 実装を `isu-python` に寄せた。
* CSS/JS/画像などの静的ファイルは Nginx から直接配信するようにした。
* 投稿一覧の N+1 を潰し、`posts` / `comments` 側のクエリを軽くした。
* `gunicorn` は `--workers 2` で運用している。
* ユーザー情報は ID 単位でメモ化している。
* ディスク増加対策として、ベンチ用の不要ファイルや MySQL binlog 周辺を整理済み。

## 5. 次の作業者への引き継ぎポイント

1. まず `make bench` を回し、現状スコアと `docs/score_history.csv` の行を確認する。
2. 続けて `make anal` を回し、`alp` と `pt-query-digest` の上位を確認する。
3. 画像周りで試すなら、以下の順で当てるのが無駄が少ない。
   * アップロード時の画像処理をさらに軽くできるか
   * `/image/[0-9]+` の生成経路にまだ DB 往復が残っていないか
   * 画像以外で大きい SQL が残っていないか

### 触らない方がよさそうだったもの
* コメントの memcached キャッシュはスコアが下がったため revert 済み。
* 画像ファイルの再利用も、見た目は良くても最終スコアは下がったため revert 済み。
