# `make setup` 実行内容の詳細解説

ISUCONにおいて「今サーバーに何がインストールされ、どう設定が変更されたか」を把握することは、後で環境が壊れた際の手戻りやボトルネック特定において非常に重要です。

先ほどの `make setup`（実体は `setup.sh`）がサーバー上で実行した処理群は、**「競技環境を安定化させ、高度な計測を可能にするための最強の土台作り」**です。具体的に行われた4つの主要プロセスを解説します。

---

### 1. `unattended-upgrades` の無効化
```text
Disabling unattended-upgrades...
Executing: /usr/lib/systemd/systemd-sysv-install disable unattended-upgrades
```
**【何をしたか】**
Ubuntuが裏側で勝手に実行する「自動セキュリティアップデート」機能を完全に停止しました。

**【なぜ必要か？】**
ISUCONのベンチマーク中にこの自動アップデートが突然走り出すと、**CPU資源とディスクI/Oを強烈に持っていかれ、原因不明の大幅なスコアダウン（またはタイムアウトによるFail）**を引き起こします。これを防ぐための絶対的な定石です。

---

### 2. `Netdata` のインストール
```text
Installing netdata...
...
Successfully installed the Netdata Agent.
```
**【何をしたか】**
リアルタイムのシステムパフォーマンス監視ツールである「Netdata」の公式リポジトリを追加し、エージェントをインストール・起動しました。

**【なぜ必要か？】**
Netdataは `http://54.150.95.245:19999` にアクセスするだけで、**「現在、CPU・メモリ・ディスクI/O・ネットワーク帯域のどれが限界に達しているか」**を1秒単位の美しいグラフィカルなUIで表示してくれます。
「アプリが遅いのか、DBが遅いのか、そもそも帯域が詰まっているのか」という大局的なボトルネックを直感的に特定するための最重要ツールです。

---

### 3. `percona-toolkit` のインストール
```text
Installing percona-toolkit...
...
Unpacking percona-toolkit (3.2.1-1) ...
```
**【何をしたか】**
強力なDB管理ツール群である `percona-toolkit` を、途中の対話プロンプト（Yes/Noの確認など）を無視する設定（`DEBIAN_FRONTEND=noninteractive`）で強制インストールしました。依存関係として `libdbi-perl` なども入っています。

**【なぜ必要か？】**
これに含まれる **`pt-query-digest`** というコマンドを使うためです。
MySQLが出力する「スロークエリログ（遅いSQLの生ログ）」はそのままでは非常に読みづらいですが、このツールを通すことで**「どのSQLが、何回呼ばれて、合計何秒かかっているか」**を綺麗にランキング形式で集計してくれます。これでSQLのインデックス不足やN+1問題を一撃で特定します。

---

### 4. 実行言語（Python）への切り替え
```text
Switching to python app...
```
**【何をしたか】**
初期状態で動いていた可能性のあるRuby実装（`isu-ruby`）を停止・無効化し、今回チューニングを行うPython実装（`isu-python`）のサービスを起動・有効化（OS再起動時にも自動起動）しました。

---

## 💡 まとめ
このスクリプトが完走したことで、サーバーは**「外的要因でスコアがブレない安定した状態」**かつ**「Nginx（alp）、MySQL（pt-query-digest）、OS全体（Netdata）の全レイヤーで計測結果が可視化される状態」**に仕上がりました。

手戻りが発生した際も、このドキュメントとGitの履歴（`setup.sh`の内容）を見れば、初期化時に何を行ったかがすぐに確認できます。
